## 主要风险

1. 战斗时间模型不一致  
   当前自动战斗和离线收益都接近按 `COMBAT_INTERVAL_MS` 结算一场。即使 `killTime` 接近 60 秒，只要能赢也可能按 1.2 秒一场发奖，经济会被放大。

2. 已装备物品可能重复穿戴  
   从已装备列表选中装备再点击替换，存在把同一装备推回背包的风险，需要让同一已装备物品穿戴操作成为 no-op。

3. 寻宝属性单位不一致  
   `magicFind` 词缀是小数百分比，但掉落率加成中又乘了 `0.01`，收益明显偏低；`goldFind`/`expFind` 则按 `1 + value` 使用。

4. 存档校验较弱  
   `importSave()` 只检查基础字段，坏 JSON、缺字段、旧版本、高版本、异常枚举等保护不足。

5. UI 自动化仍偏薄  
   当前 UI 只有冒烟测试，尚未覆盖保存、离线领取、训练、强化、一键穿戴、切层失败提示等核心交互。

## 下一步建议

优先做 P0 稳定性补丁，不建议立刻进入第二阶段。

1. 统一战斗时间模型  
   让线上挂机和离线收益都基于 `killTime` 或实际战斗耗时扣时间，避免经济膨胀。

2. 修装备状态一致性  
   `equip()` 遇到同一已装备物品时直接返回，不产生背包副本，并补防复制测试。

3. 补存档可靠性  
   增加 `pagehide`/`visibilitychange` 即时保存，统一离线时间口径，并补坏档/版本不匹配测试。

4. 补体验提示  
   手动切层失败时给出明确提示；战斗日志显示包含金币/经验获取加成后的实际收益。

5. 做真实浏览器验收  
   按链路检查：开始挂机 → 掉装 → 对比 → 穿戴 → DPS/EHP 变化 → 保存刷新 → 离线报告 → 一键穿戴。

- 存档采用单槽 `localStorage`，复杂迁移和多槽位延后。
