# 装备排序和筛选面板开发记录

## 目标

本阶段为背包增加装备排序和筛选面板，让玩家可以按品质、部位、评分、是否更优、锁定状态快速整理装备，降低背包数量增长后的查找和分解成本。

## 范围边界

- 排序和筛选只影响背包展示顺序，不改变装备实体、穿戴状态、掉落结果和存档结构。
- 核心判断逻辑应优先放在 `core/item/` 纯函数中，组件只负责展示筛选条件和触发交互。
- Store 负责保存当前筛选条件、排序字段和升降序，不在组件内散落业务状态。
- 本阶段不处理虚拟滚动、批量标签管理和装备搜索框，除非背包容量扩大导致性能风险提前暴露。

## 任务拆解

主智能体负责实现和集成：

- 梳理现有背包展示、装备评分、更优高亮、锁定状态和自动拾取过滤逻辑。
- 设计 `EquipmentSortField`、`EquipmentFilterState`、`EquipmentSortDirection` 等领域类型。
- 新增或扩展装备列表纯函数，支持按品质、评分、部位、是否更优、锁定状态排序和筛选。
- 在 Pinia 背包状态中接入筛选条件、排序字段、升降序、重置筛选动作。
- 在背包面板中新增排序和筛选入口，移动端优先使用抽屉或折叠面板。
- 补充核心排序筛选测试，覆盖空列表、多条件组合、稳定排序和重置行为。

子智能体 A 负责核心规则复核：

- 确认排序字段不会依赖 Vue、Pinia、DOM 或浏览器副作用。
- 复核品质顺序、部位顺序、评分缺失值、戒指等多槽位装备的比较规则。
- 关注筛选结果和自动分解保护规则之间的认知一致性。

子智能体 B 负责界面与交互复核：

- 复核移动端面板高度、触控目标尺寸、筛选角标和空结果状态。
- 确认排序按钮、筛选按钮、重置按钮和升降序按钮在深色背包界面中可识别。
- 确认已启用筛选时有清晰状态反馈，避免玩家误以为装备丢失。

子智能体 C 负责美术资源跟进：

- 根据本阶段美术需求清单整理资源命名、规格和优先级。
- 在资源未完成前，明确 CSS 占位策略和后续替换点。

## 推荐交互流程

1. 玩家打开背包后，默认按当前既有顺序展示装备。
2. 点击排序按钮可选择评分、品质、部位、获得顺序、是否更优。
3. 点击升降序按钮可在升序和降序之间切换。
4. 点击筛选按钮打开筛选面板，可选择品质、部位、是否更优、锁定状态。
5. 启用任意筛选后，背包入口或筛选按钮展示已启用角标。
6. 筛选结果为空时展示空结果插画和重置筛选入口。
7. 点击重置筛选后清空所有筛选条件，排序条件可保留或按产品确认恢复默认。

## 验收标准

- 背包装备可按评分、品质、部位、是否更优进行排序。
- 排序支持升序和降序切换，切换后列表稳定且不丢失装备。
- 筛选支持品质、部位、更优装备、锁定状态。
- 多个筛选条件组合时结果符合交集规则。
- 启用筛选后有角标提示，空结果有明确提示和重置入口。
- 重置筛选能恢复完整背包列表。
- 相关核心逻辑有 Vitest 覆盖，类型检查通过。

## 风险与取舍

- 品质和部位属于枚举顺序，必须集中定义，避免 UI 和核心逻辑排序不一致。
- “更优装备”依赖评分模型，后续流派权重变化时需要同步复核。
- 筛选后执行分解容易造成误解，分解确认弹窗仍应展示最终将分解和被保护装备。
- 初版优先保证可读、可测、可维护，不引入复杂动画。
