# 2026-05-09 设计对齐记录

## 背景

当前核心循环已经能跑，需要根据实际开发情况回看原始 GDD 和开发文档，确认设计方向是否合理，并区分当前必须做的核心系统与未来扩展内容。

## 关键判断

- 游戏方向没有跑偏，仍然是“装备 Build 爽感 + 放置轻节奏”。
- 当前已经完成的是可测试核心循环原型，不是正式完整产品。
- 下一阶段不能直接铺强化、套装、转生、多职业等长期系统。
- 当前最需要补的是“构筑选择、推层目标、掉落反馈”。

## 当前必须优先做的核心系统

这些系统决定游戏能不能从原型走向正式可玩：

1. 构筑评分系统
   - 解决通用评分压扁流派的问题。
   - 让暴击、攻速、坦克等装备偏好真正影响更优判断。

2. 推层目标系统
   - 让玩家知道该挂机哪层、挑战哪层、为什么失败、为什么收益降低。
   - 把后台层数逻辑转化为可理解的目标。

3. 掉落反馈系统
   - 强化传说掉落、离线领取、背包满损失、层数突破等关键反馈。
   - 不做全量美术替换，先做少量高价值反馈。

4. 基础内容差异
   - 每 10 层建立轻量 Boss 或阶段目标。
   - 增强怪物和层数差异，避免纯数值堆叠。

5. 稳定性和测试
   - 保持核心逻辑测试。
   - 逐步补组件交互测试，覆盖关键弹窗和筛选/导入流程。

## 未来扩展补充

这些内容应在核心目标清晰、正式可玩稳定后再推进：

- 完整天赋树。
- 装备强化高等级系统。
- 传说特效扩展。
- 远古装备。
- 套装。
- 转生或轮回。
- 多职业。
- 排行榜和装备展示。
- 完整音效系统。
- 完整 GSAP 动效体系。
- 更严格的离线收益防作弊策略。

## 设计红线

- 不让背包整理变成主玩法。
- 不让通用装备评分成为唯一答案。
- 不在构筑目标明确前扩展大量长期系统。
- 不用正式资源替换掩盖玩法目标不清的问题。
- 不牺牲 core 纯逻辑和测试边界。
