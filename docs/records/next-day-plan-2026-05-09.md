## 明天从这里开始

第一件事做完整存档服务。

目标是把当前 Pinia persistedstate 的基础存储，升级为更可控的本地存档能力：

```text
当前状态 -> 生成版本化快照 -> localForage 保存 -> 导出/导入 -> 版本迁移 -> 恢复状态
```

建议先做最小可用版本，不急着做复杂 UI。

## P0 任务

1. 设计存档快照类型
   - 新增或整理 `src/types/save.ts`。
   - 快照至少包含版本号、保存时间、玩家、背包、设置、背包视图、存档元信息。
   - 不建议持久化 `isAutoFighting`，刷新后自动继续挂机容易产生状态误解。
   - `combat` 是否保存当前层数和最高解锁层需要先确认现有 store 字段语义。

2. 建立存档服务层
   - 推荐新增 `src/services/saveService.ts` 或 `src/core/save/` 纯逻辑 + `stores/save` 组合。
   - localForage 读写、导入导出放在 service/store 层，不放进 `core`。
   - 版本迁移函数保持纯逻辑，便于测试。

3. 接入 store 行为
   - 在 `stores/save.ts` 中增加创建快照、恢复快照、导出存档、导入存档入口。
   - 恢复快照时注意不要覆盖未包含的未来字段。
   - 导入失败要返回明确错误，不要吞掉异常。

4. 补测试
   - 优先测快照创建和迁移纯逻辑。
   - 再测 `save` store 导入导出行为。
   - localForage 可以先通过 mock 或 service 抽象隔离，避免测试依赖真实浏览器存储。

## P1 任务

1. 给存档导入导出补 UI 入口
   - 可以先放在右侧面板或后续独立设置面板。
   - 如果今天只做底层服务，UI 可以后置。

2. 清理 `RewardReport.vue` 的低风险 core 工具引用
   - 将 `formatDuration` 迁到 `src/utils/format.ts` 或新增离线展示 composable。
   - 这是小任务，可以作为存档服务之后的收尾。

3. 评估 `combat` 持久化
   - 建议保存当前层数、最高解锁层和最近战力结果。
   - 不建议保存自动战斗运行态。
